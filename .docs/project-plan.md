# Error Free Text — план проекта

Статус документа: живой черновик. План уточняется по мере поступления частей технического задания.

## 1. Цель продукта

Разработать веб-приложение **Error Free Text**, которое принимает пользовательский текст и выполняет его корректировку в рамках создаваемой задачи.

## 2. Подтверждённые требования

- Архитектура приложения: монолит.
- Внешний интерфейс: REST API.
- Корректировка запускается через создание задачи.
- Запрос на создание задачи содержит исходный текст и язык текста.
- Полученные данные сохраняются в базе данных.
- Новая задача сохраняется со статусом `NEW` (рабочее обозначение статуса до уточнения точного значения в контракте).
- В ответ на создание задачи возвращается её идентификатор.
- Результат корректировки запрашивается по идентификатору задачи.
- Для завершённой задачи ответ содержит статус и скорректированный текст.
- Для выполняющейся задачи ответ содержит только статус.
- Для задачи, завершившейся с ошибкой, ответ содержит статус и описание ошибки.
- Для корректировки используется метод `POST checkTexts` API Яндекс Спеллера.
- Текст размером более 10 000 символов разбивается на части, допустимые для API Яндекса.
- При наличии цифр включается опция `IGNORE_DIGITS`.
- При наличии URL включается опция `IGNORE_URLS`.
- Опции `FIND_REPEAT_WORDS` и `IGNORE_CAPITALIZATION` всегда выключены.
- При ошибке обращения к API Яндекса задача переводится в ошибочное состояние, а её обработка прекращается.
- Допустимые значения языка во входном запросе: `en` и `ru`.
- Текст должен содержать не менее трёх символов.
- Текст не может состоять только из цифр и специальных символов.
- API проектируется с учётом принципов REST.
- Приложение должно логировать значимые события и обрабатывать ошибки.
- Ошибки обработки входящих запросов возвращаются в едином формате с сообщением, внутренним кодом, временем и путём запроса.
- Unit-тесты обязательны; интеграционные тесты опциональны.
- Код должен соответствовать принципам Clean Code и Java style.

## 3. Предварительные границы системы

На текущем этапе в систему входят:

- приём и проверка REST-запроса;
- создание и хранение задачи на корректировку;
- обработка текста по правилам, которые будут определены в ТЗ;
- формирование успешного ответа;
- единообразная обработка ошибок.

Границы будут дополнены после уточнения жизненного цикла задачи, пользователей, истории запросов и внешних интеграций.

## 4. REST API

### 4.1. Создание задачи на корректировку

Статус контракта: предварительный — состав данных подтверждён, URI и точные форматы ещё требуют уточнения.

Предполагаемый запрос:

```http
POST /api/v1/correction-tasks
Content-Type: application/json
```

```json
{
  "text": "Текст для корректировки",
  "language": "ru"
}
```

Предполагаемый успешный ответ:

```http
HTTP/1.1 201 Created
```

```json
{
  "id": 1
}
```

Алгоритм операции:

1. Принять исходный текст и язык.
2. Проверить обязательные поля и ограничения входных данных.
3. Создать задачу со статусом новой задачи.
4. Сохранить задачу в базе данных.
5. Вернуть идентификатор сохранённой задачи.

Атомарность: идентификатор возвращается только после успешного сохранения задачи. При ошибке сохранения успешный ответ не формируется.

#### Валидация запроса

Запрос проходит валидацию до создания задачи и обращения к базе данных.

| Поле | Правило |
|---|---|
| `language` | Обязательное значение, строго `en` или `ru` |
| `text` | Обязательное значение длиной не менее 3 символов |
| `text` | Должен содержать хотя бы одну Unicode-букву |

Проверка наличия хотя бы одной буквы формализует запрет на текст, состоящий только из цифр, специальных символов и пробелов.

При нарушении любого правила задача не создаётся. Предварительный HTTP-ответ — `400 Bad Request` в едином формате из раздела 4.3; конкретный `errorCode` для ошибок валидации ещё не определён.

### 4.2. Получение состояния и результата задачи

Статус контракта: предварительный — принцип формирования ответа подтверждён, URI, HTTP-коды и точные значения статусов требуют уточнения.

Предполагаемый запрос:

```http
GET /api/v1/correction-tasks/{id}
```

Если задача успешно завершена:

```json
{
  "status": "COMPLETED",
  "correctedText": "Исправленный текст"
}
```

Если задача новая или находится в процессе обработки:

```json
{
  "status": "PROCESSING"
}
```

Если обработка завершилась с ошибкой:

```json
{
  "status": "FAILED",
  "error": "Описание ошибки"
}
```

Рабочие названия `COMPLETED`, `PROCESSING` и `FAILED` используются только для планирования до получения точного перечня статусов из ТЗ.

Правила формирования ответа:

- поле `status` присутствует для любой найденной задачи;
- `correctedText` возвращается только после успешного завершения;
- `error` возвращается только после завершения с ошибкой;
- незаполненные условные поля не включаются в JSON-ответ.

### 4.3. Единый ответ об ошибке HTTP-запроса

При ошибке обработки входящего REST-запроса приложение возвращает JSON следующей структуры:

```json
{
  "errorMessage": "Task with id: 44bd78dc-d08c-41c6-b87d-fb82046bd470 not found",
  "errorCode": 40401,
  "timestamp": "2005-08-09T18:31:42.201",
  "path": "/tasks"
}
```

В исходном примере исправлена закрывающая кавычка у `timestamp`, чтобы пример являлся валидным JSON.

| Поле | Назначение | Предварительный тип |
|---|---|---|
| `errorMessage` | Читаемое описание ошибки | string |
| `errorCode` | Внутренний прикладной код ошибки | integer |
| `timestamp` | Момент формирования ответа в ISO-8601 | string |
| `path` | Путь фактического HTTP-запроса | string |

Для отсутствующей задачи прикладной код `40401` подтверждён примером, а HTTP-статус `404 Not Found` предварительно выбран по REST-семантике. Остальные соответствия HTTP-статусов и прикладных кодов требуют определения.

Этот формат относится к ошибкам обработки HTTP-запроса и не заменяет представление результата задачи в состоянии ошибки из раздела 4.2: найденная задача в состоянии `FAILED` возвращается как состояние ресурса.

## 5. Предварительная модель данных

### CorrectionTask

| Поле | Назначение | Статус определения |
|---|---|---|
| `id` | Идентификатор задачи | Предварительно UUID на основании примера ошибки |
| `sourceText` | Исходный текст для корректировки | Подтверждено |
| `language` | Язык исходного текста: `en` или `ru` | Подтверждено |
| `status` | Текущее состояние задачи | Начальное состояние подтверждено |
| `correctedText` | Результат успешной корректировки | Подтверждено, допускает отсутствие до завершения |
| `errorDescription` | Описание ошибки обработки | Подтверждено, допускает отсутствие при других состояниях |

Необходимость дат создания/обновления и дополнительной диагностической информации будет определена следующими требованиями.

### Жизненный цикл задачи

Предварительные переходы состояний:

```text
NEW → PROCESSING → COMPLETED
                 ↘ FAILED
```

Точные значения enum и допустимость прямых переходов `NEW → COMPLETED` или `NEW → FAILED` ещё не определены.

## 6. Интеграция с Яндекс Спеллером

### 6.1. Внешний контракт

- метод: `POST checkTexts`;
- предполагаемый JSON endpoint: `https://speller.yandex.net/services/spellservice.json/checkTexts`;
- кодировка запроса: UTF-8;
- `text`: массив фрагментов текста;
- `lang`: язык проверки;
- `options`: сумма числовых значений включённых опций;
- максимальный размер текста в одном POST-запросе: 10 000 символов суммарно;
- ответ содержит отдельный массив найденных ошибок для каждого переданного фрагмента.

Источники: [метод `checkTexts`](https://yandex.ru/dev/speller/doc/ru/reference/checkTexts), [настройки Яндекс Спеллера](https://yandex.ru/dev/speller/doc/ru/reference/speller-options), [коды найденных ошибок](https://yandex.ru/dev/speller/doc/ru/reference/error-codes).

Хотя Яндекс также поддерживает `uk` и комбинации языков, контракт нашего приложения принимает только одиночные значения `en` и `ru`. Они передаются в Яндекс Спеллер без преобразования.

### 6.2. Расчёт options

Опции вычисляются для исходного текста задачи до отправки запросов и применяются ко всем его частям.

| Условие | Опция | Значение |
|---|---|---:|
| В тексте есть цифры | `IGNORE_DIGITS` | 2 |
| В тексте есть URL | `IGNORE_URLS` | 4 |
| Всегда выключена | `FIND_REPEAT_WORDS` | 8 |
| Всегда выключена | `IGNORE_CAPITALIZATION` | 512 |

Возможные итоговые значения `options`:

| Цифры | URL | `options` |
|---|---|---:|
| Нет | Нет | 0 |
| Да | Нет | 2 |
| Нет | Да | 4 |
| Да | Да | 6 |

Выключенные опции никогда не добавляются к итоговому значению.

### 6.3. Обработка большого текста

Предварительный алгоритм:

1. Проанализировать весь исходный текст и вычислить единое значение `options`.
2. Разбить текст на упорядоченные фрагменты так, чтобы суммарный объём `text` одного запроса не превышал 10 000 символов.
3. По возможности выбирать границы между словами и не разрывать URL.
4. Последовательно отправлять фрагменты через `POST checkTexts`, сохраняя их исходный порядок.
5. Для каждого фрагмента применить исправления по `pos` и `len` из ответа Яндекса.
6. Соединить исправленные фрагменты без потери или добавления разделителей.
7. После успешной обработки всех частей сохранить итоговый текст и перевести задачу в состояние успешного завершения.

Последовательная отправка выбрана предварительно, поскольку при первой ошибке обработка должна быть прекращена. Параллельная отправка потребует отмены уже запущенных запросов и пока не предполагается.

### 6.4. Обработка ответа

Яндекс возвращает позицию и длину ошибочного фрагмента, исходное слово и массив вариантов замены. Требуется определить правило выбора варианта; предварительно предполагается использовать первый предложенный вариант, а при пустом массиве подсказок оставлять исходный текст без изменения.

Исправления внутри одного фрагмента следует применять справа налево либо собирать результат по диапазонам исходного текста, чтобы изменение длины одной замены не сдвигало позиции следующих исправлений.

Коды `1`–`4` внутри элементов ответа описывают тип найденной орфографической проблемы и сами по себе не означают сбой обращения к API.

### 6.5. Ошибка интеграции

К ошибке обращения предварительно относятся:

- ошибка соединения или тайм-аут;
- неуспешный HTTP-ответ внешнего API;
- ответ, который невозможно разобрать или сопоставить отправленным фрагментам.

При первой такой ошибке необходимо:

1. прекратить отправку оставшихся частей;
2. не формировать частично исправленный результат как успешный;
3. сохранить описание ошибки;
4. перевести задачу в ошибочное состояние.

Политика повторных попыток в ТЗ пока не определена.

## 10. Критерии готовности плана

План можно считать завершённым, когда:

- все требования из ТЗ отражены в документе;
- по каждому требованию есть реализационная задача и способ проверки;
- закрыты вопросы, влияющие на архитектуру и REST-контракт;
- определены этапы реализации и порядок зависимых работ;
- обязательный набор unit-тестов определён и реализуем изолированно;
- определены единая модель ошибок и минимальная политика логирования;
- составлен итоговый чек-лист приёмки.
