# Error Free Text — план проекта

## 1. Цель продукта

Разработать веб-приложение **Error Free Text**, которое принимает пользовательский текст и выполняет его корректировку в рамках создаваемой задачи.

Правило источника истины: раздел 2 фиксирует исходные требования, разделы 3–5 содержат их нормативную детализацию, раздел 6 отражает только статус реализации, а раздел 7 — выбранные технические решения, не заданные исходным ТЗ. Если краткая формулировка требования допускает разные трактовки, применяется соответствующее правило из разделов 3–5.

## 2. Исходные подтверждённые требования

- Архитектура приложения: монолит.
- Внешний интерфейс: REST API.
- Корректировка запускается через создание задачи.
- Запрос на создание задачи содержит исходный текст и язык текста.
- Полученные данные сохраняются в базе данных.
- Новая задача сохраняется со статусом `NEW`.
- В ответ на создание задачи возвращается её идентификатор.
- Результат корректировки запрашивается по идентификатору задачи.
- Для завершённой задачи ответ содержит статус и скорректированный текст.
- Для выполняющейся задачи ответ содержит только статус.
- Для задачи, завершившейся с ошибкой, ответ содержит статус и описание ошибки.
- Для корректировки используется метод `POST checkTexts` API Яндекс Спеллера.
- Текст размером более 10 000 символов разбивается на части, допустимые для API Яндекса.
- При наличии цифр включается опция `IGNORE_DIGITS`.
- При наличии URL включается опция `IGNORE_URLS`.
- Опции `FIND_REPEAT_WORDS` и `IGNORE_CAPITALIZATION` всегда выключены.
- При ошибке обращения к API Яндекса задача переводится в ошибочное состояние, а её обработка прекращается.
- Допустимые значения языка во входном запросе: `en` и `ru`.
- Текст должен содержать не менее трёх символов.
- Текст не может состоять только из цифр и специальных символов.
- API проектируется с учётом принципов REST.
- Приложение должно логировать значимые события и обрабатывать ошибки.
- Ошибки обработки входящих запросов возвращаются в едином формате с сообщением, внутренним кодом, временем и путём запроса.
- Unit-тесты обязательны; интеграционные тесты опциональны.
- Код должен соответствовать принципам Clean Code и Java style.
- Фоновый запуск корректировки выполняется Scheduler-ом внутри приложения.
- Язык реализации: Java 17 или новее.
- Фреймворк: Spring Boot 3.
- Система сборки: Gradle.
- База данных: PostgreSQL.
- Контейнеризация: Docker и Docker Compose.

## 3. REST API

### 3.1. Создание задачи на корректировку

Запрос:

```http
POST /tasks
Content-Type: application/json
```

```json
{
  "text": "Текст для корректировки",
  "language": "ru"
}
```

Успешный ответ:

```http
HTTP/1.1 201 Created
```

```json
{
  "id": "44bd78dc-d08c-41c6-b87d-fb82046bd470"
}
```

Заголовок `Location` не возвращается: идентификатора в теле ответа достаточно для последующего запроса задачи. После успешной валидации задача сохраняется со статусом `NEW`; идентификатор возвращается только после успешного сохранения. При ошибке сохранения успешный ответ не формируется.

#### Валидация запроса

Запрос проходит валидацию до создания задачи и обращения к базе данных.

| Поле | Правило |
|---|---|
| `language` | Обязательное значение, строго `en` или `ru` |
| `text` | После `strip()` содержит не менее 3 Unicode code points |
| `text` | Должен содержать хотя бы одну Unicode-букву |

Проверка наличия хотя бы одной буквы формализует запрет на текст, состоящий только из цифр, специальных символов и пробелов.

`strip()` и подсчёт code points используются только для валидации. Исходная строка сохраняется и передаётся на корректировку без удаления окружающих пробелов.

При нарушении любого правила задача не создаётся. Приложение возвращает `400 Bad Request` с кодом `40001` в едином формате из раздела 3.3.

### 3.2. Получение состояния и результата задачи

Запрос:

```http
GET /tasks/{id}
```

Для любой существующей задачи endpoint возвращает `200 OK`. Поле `status` обязательно, а состав остальных полей зависит от состояния:

| `status` | Поля ответа |
|---|---|
| `NEW` | `status` |
| `PROCESSING` | `status` |
| `COMPLETED` | `status`, `correctedText` |
| `FAILED` | `status`, `errorMessage` |

Незаполненные условные поля в JSON-ответ не включаются.

### 3.3. Единый ответ об ошибке HTTP-запроса

При ошибке обработки входящего REST-запроса приложение возвращает JSON следующей структуры:

```json
{
  "errorMessage": "Task with id: 44bd78dc-d08c-41c6-b87d-fb82046bd470 not found",
  "errorCode": 40401,
  "timestamp": "2005-08-09T18:31:42.201",
  "path": "/tasks/44bd78dc-d08c-41c6-b87d-fb82046bd470"
}
```

| Поле | Назначение | Тип |
|---|---|---|
| `errorMessage` | Читаемое описание ошибки | string |
| `errorCode` | Внутренний прикладной код ошибки | integer |
| `timestamp` | Момент формирования ответа в ISO-8601 | string |
| `path` | Фактический путь HTTP-запроса, например `/tasks/{id}` | string |

Каталог ошибок публичного API:

| HTTP-статус | `errorCode` | Назначение |
|---:|---:|---|
| 400 | `40001` | Данные корректного запроса не прошли валидацию |
| 400 | `40002` | Некорректный JSON, тип поля или формат UUID |
| 404 | `40401` | Задача не найдена |
| 405 | `40501` | HTTP-метод не поддерживается |
| 415 | `41501` | `Content-Type` не поддерживается |
| 500 | `50001` | Непредвиденная внутренняя ошибка |

Формирование ответов централизуется в одном `@RestControllerAdvice`. Он обрабатывает ошибки валидации и разбора запроса, `TaskNotFoundException`, соответствующие исключения Spring MVC и непредвиденные исключения.

Для `50001` клиенту возвращается безопасное общее сообщение без stack trace и внутренних деталей; полное исключение записывается в лог.

Этот формат относится к ошибкам обработки HTTP-запроса и не заменяет представление результата задачи в состоянии ошибки из раздела 3.2: найденная задача в состоянии `FAILED` возвращается как состояние ресурса.

## 4. Модель данных

### CorrectionTask

| Поле | Назначение |
|---|---|
| `id` | Идентификатор задачи UUID v4 |
| `sourceText` | Исходный текст для корректировки |
| `language` | Язык исходного текста: `en` или `ru` |
| `status` | `NEW`, `PROCESSING`, `COMPLETED` или `FAILED` |
| `correctedText` | Результат успешной корректировки; отсутствует до завершения и при ошибке |
| `errorMessage` | Безопасное описание ошибки; присутствует только в состоянии `FAILED` |
| `createdAt` | Момент создания задачи в UTC; не возвращается через REST |
| `updatedAt` | Момент последнего изменения задачи в UTC; не возвращается через REST |

Идентификатор генерируется через `UUID.randomUUID()`, а его уникальность обеспечивается первичным ключом PostgreSQL. Исходный и исправленный тексты хранятся в колонках типа `TEXT`.

В Java даты представлены типом `Instant`, в БД — типом с поддержкой часового пояса. `updatedAt` изменяется при обновлении статуса, результата или ошибки.

### Жизненный цикл задачи

Допустимые переходы состояний:

```text
NEW → PROCESSING → COMPLETED
                 ↘ FAILED
```

`COMPLETED` и `FAILED` являются терминальными состояниями. Статусы хранятся строками: Java enum `TaskStatus`, `EnumType.STRING` в JPA и те же значения в REST API.

### Механизм запуска обработки

В приложении работает Scheduler, который выбирает сохранённые задачи `NEW` и запускает их корректировку. Благодаря хранению очереди в БД задачи `NEW` остаются доступными после перезапуска приложения.

Scheduler запускается с настраиваемым `fixedDelay` 1 секунда, выбирает самую старую задачу `NEW` по `createdAt` и обрабатывает только одну задачу одновременно.

Интервал задаётся свойством `task.processing.polling-interval` в `application.yml`. Для выборки создаётся составной индекс по `(status, created_at)`.

При запуске приложения все оставшиеся задачи `PROCESSING` одной транзакцией переводятся в `FAILED` с сообщением `Text correction was interrupted`. Количество восстановленных задач записывается в лог. Возврат в `NEW` не выполняется, поскольку означал бы автоматическую повторную попытку.

### Политика логирования

Используются SLF4J и стандартный Logback без дополнительной инфраструктуры.

`INFO`:

- создание задачи: `taskId`, язык, длина текста;
- начало обработки: `taskId`, количество частей, `options`;
- успешное завершение: `taskId`, длительность обработки;
- запуск приложения и количество оставшихся `PROCESSING`, переведённых в `FAILED`.

`WARN`:

- тайм-аут или ожидаемая ошибка Яндекс Спеллера с `taskId`;
- некорректный ответ внешнего API с `taskId`;
- HTTP-ошибки `40401`, `40501`, `41501` с `errorCode`, `path` и `taskId`, если он известен.

`ERROR`:

- непредвиденное исключение со stack trace и `taskId`, если он известен;
- ошибка доступа к БД;
- HTTP-ошибка `50001` со stack trace.

Ошибки `40001` и `40002` не логируются, чтобы не создавать шум из-за некорректных клиентских данных. Исходный и исправленный текст, а также тела запросов и ответов Яндекса не логируются.

### Нефункциональные границы

- максимальная длина входного текста на уровне приложения не устанавливается;
- аутентификация и разделение пользователей не входят в scope; любой клиент со знанием UUID может запросить задачу;
- Actuator и прикладные метрики не входят в scope; наблюдаемость обеспечивается логированием;
- кэширование, несколько обработчиков задач и нагрузочные тесты не входят в scope;
- время выполнения задачи не гарантируется и зависит от размера текста и времени ответа Яндекс Спеллера.

## 5. Интеграция с Яндекс Спеллером

### 5.1. Внешний контракт

- HTTP-клиент: синхронный Spring `RestClient` из `spring-boot-starter-web`;
- метод: `POST checkTexts`;
- JSON endpoint: `https://speller.yandex.net/services/spellservice.json/checkTexts`;
- кодировка запроса: UTF-8;
- `text`: массив фрагментов текста;
- `lang`: язык проверки;
- `options`: сумма числовых значений включённых опций;
- максимальный размер одного элемента массива `text`: 10 000 UTF-16 units по Java `String.length()`;
- ответ содержит отдельный массив найденных ошибок для каждого переданного фрагмента.

Источники: [метод `checkTexts`](https://yandex.ru/dev/speller/doc/ru/reference/checkTexts), [настройки Яндекс Спеллера](https://yandex.ru/dev/speller/doc/ru/reference/speller-options), [коды найденных ошибок](https://yandex.ru/dev/speller/doc/ru/reference/error-codes).

Контракт нашего приложения принимает только одиночные значения `en` и `ru`. Они передаются в Яндекс Спеллер без преобразования.

### 5.2. Расчёт options

Опции вычисляются за один линейный проход по исходному тексту задачи до отправки запросов и применяются ко всем его частям. Цифры обнаруживаются по Unicode code points через `Character.isDigit`.

URL обнаруживается самим приложением до разбиения текста. Поддерживаются:

- `http://`, `https://` и `ftp://` с необязательными портом, path, query и fragment;
- адреса, начинающиеся с `www.`;
- домены без протокола с любой синтаксически корректной доменной зоной;
- Unicode-домены с нормализацией доменного имени через `IDN.toASCII()`;
- IPv4, IPv6 и `localhost` при наличии протокола.

Email и имена файлов сами по себе не считаются URL для условия включения опции. Детектор не содержит фиксированного списка доменных зон: кандидаты извлекаются заранее скомпилированным `Pattern` и проверяются синтаксически.

| Условие | Опция | Значение |
|---|---|---:|
| В тексте есть цифры | `IGNORE_DIGITS` | 2 |
| В тексте есть URL | `IGNORE_URLS` | 4 |
| Всегда выключена | `FIND_REPEAT_WORDS` | 8 |
| Всегда выключена | `IGNORE_CAPITALIZATION` | 512 |

Итоговое значение `options` равно сумме включённых опций и может быть равно 0, 2, 4 или 6. Выключенные опции в сумму не добавляются.

### 5.3. Обработка большого текста

Алгоритм:

1. Вычислить для исходного текста единое значение `options` по правилам раздела 5.2.
2. Разбить текст на упорядоченные фрагменты, `String.length()` каждого из которых не превышает 10 000.
3. По возможности выбирать границы по переносу строки или пробелу и не разрывать слова и URL.
4. Передать все фрагменты одним упорядоченным массивом `text` в одном `POST checkTexts`.
5. Для каждого фрагмента применить исправления по `pos` и `len` из ответа Яндекса.
6. Соединить исправленные фрагменты без потери или добавления разделителей.
7. После успешной обработки всех частей сохранить итоговый текст и перевести задачу в состояние успешного завершения.

Количество и порядок внутренних массивов внешнего ответа должны совпадать с количеством и порядком отправленных фрагментов.

Граница фрагмента никогда не проводится между старшим и младшим элементами surrogate pair. Благодаря этому emoji и другие дополнительные Unicode-символы сохраняются целиком. Этот подсчёт также совместим с наблюдаемыми значениями `pos` и `len` Яндекс Спеллера для текста с emoji.

Если до лимита нет безопасной границы, применяется жёсткое разбиение: фрагмент заканчивается на максимально допустимой позиции, даже если она находится внутри одного длинного слова или URL. При этом каждый символ исходного текста входит ровно в один фрагмент и отправляется Яндекс Спеллеру; пропуск URL не допускается. Единственное смещение границы выполняется при необходимости сохранить целостность surrogate pair.

Единое значение `options` применяется ко всему массиву фрагментов. Поэтому при наличии URL `IGNORE_URLS` действует и для фрагмента, содержащего только часть длинного URL. Такая часть может перестать распознаваться внешним API как полноценный URL, однако сохранить URL длиннее 10 000 одним фрагментом при соблюдении лимита невозможно.

### 5.4. Обработка ответа

Яндекс Спеллер является единственным источником определения опечаток: приложение не выполняет дополнительную классификацию или самостоятельный поиск ошибок. Ошибкой считается каждый элемент, возвращённый `checkTexts` с позицией `pos`, длиной `len` и массивом подсказок `s`, с учётом переданных опций.

Яндекс возвращает позицию и длину ошибочного фрагмента, исходное слово и массив вариантов замены. Для автоматической коррекции используется первая подсказка после удаления окружающих пробелов: `s[0].strip()`. Внутренние пробелы сохраняются. Если `s` отсутствует, пуст или результат `strip()` пуст, наличие элемента ошибки всё равно означает, что Яндекс распознал проблему, но не предложил применимую замену: исходный фрагмент сохраняется без изменений, а обработка задачи продолжается.

Исправления внутри одного фрагмента следует применять справа налево либо собирать результат по диапазонам исходного текста, чтобы изменение длины одной замены не сдвигало позиции следующих исправлений.

Коды `1`–`4` внутри элементов ответа описывают тип найденной орфографической проблемы и сами по себе не означают сбой обращения к API.

### 5.5. Ошибки обработки и интеграции

Первая ошибка из следующей таблицы прекращает обработку: успешный результат не формируется, задача переводится в `FAILED`, а в `error_message` сохраняется соответствующее безопасное сообщение.

| Причина | `errorMessage` |
|---|---|
| Тайм-аут внешнего запроса | `Text correction service request timed out` |
| Ошибка соединения или недоступность сервиса | `Text correction service is unavailable` |
| Неуспешный HTTP-ответ внешнего сервиса | `Text correction service returned an error` |
| Некорректная или неразбираемая структура ответа | `Text correction service returned an invalid response` |
| Ошибка применения исправлений или сборки частей | `Failed to process text correction result` |
| Другая непредвиденная ошибка обработки | `Unexpected text correction error` |
| Обработка прервана остановкой приложения | `Text correction was interrupted` |

Исходное исключение, stack trace и технические детали записываются только в лог с идентификатором задачи. Случаи успешного ответа без применимой замены обрабатываются по правилам раздела 5.4 и не переводят задачу в `FAILED`.

Если недоступна собственная база данных, приложение не может гарантированно сохранить `FAILED` и `error_message`; такая инфраструктурная ошибка фиксируется в логах. Автоматическое восстановление зависшей задачи выходит за текущие требования.

Автоматические повторные попытки не выполняются. Ошибка единственного внешнего вызова переводит задачу в `FAILED`. Для повторной обработки пользователь создаёт новую задачу.

Тайм-ауты одного вызова `checkTexts`:

- установка соединения: 3 секунды;
- ожидание ответа: 10 секунд.

Это значения по умолчанию, вынесенные в `application.yml` и переопределяемые переменными окружения. Превышение любого тайм-аута обрабатывается как ошибка внешнего вызова и переводит задачу в `FAILED`.

### 5.6. Суточные ограничения и условия использования

По условиям Яндекс Спеллера сервис устанавливает следующие ограничения:

- до 10 000 обращений в сутки;
- до 10 миллионов символов проверяемого текста в сутки.

Превышение лимита обрабатывается как неуспешный ответ внешнего сервиса: текущая задача переводится в `FAILED`.

Локальный счётчик квот и собственный rate limiter не входят в требования тестового задания. Ограничения документируются в README как эксплуатационное свойство внешней интеграции.
